This specification defines the OSLC Product Lifecycle Management (PLM) domain, a RESTful web services interface for the PLM v2 resources and relationships between those and related resources such as product change requests, activities, tasks, requirements or test cases. To support these scenarios, this specification defines a set of HTTP-based RESTful interfaces in terms of HTTP methods: GET, POST, PUT and DELETE, as well as HTTP response codes, content type handling and resource formats.

Introduction

This specification defines a RESTful web services interface for the Product Lifecycle Management (PLM) domain. This domain addresses the management of physical and logical designs of products and parts, and relationships with other OSLC resources such as requirements, testing resources and change requests. To support these scenarios, this specification defines a set of HTTP-based RESTful interfaces in terms of HTTP methods: GET, POST, PUT and DELETE, HTTP response codes, content type handling and resource formats.

The intent of this specification is to define the capabilities needed to support integration scenarios and not to provide a comprehensive interface to Product Lifecycle Management.

This specification is a [[!OSLCCore3]] compliant specification, and as such most of its content are references to [[!OSLCCore3]].

Terminology

PLM Client
An implementation of the OSLC PLM specifications as a client. OSLC PLM Clients consume services provided by PLM servers.
PLM Server
A server implementing the OSLC PLM domain specifications. OSLC PLM clients consume services provided by PLM Servers. The use of the terms Client and Server are intended to distinguish typical consumers and providers of OSLC resources in a distributed environment based on REST. A particular application component could be a client for some OSLC domain services and a server for the same or another domain.

Overview

This specification defines a vocabulary for OSLC Product Lifecycle Management resources. The intent is to define resources needed to support common integration scenarios and not to provide a comprehensive definition of product lifecycle management resources.

The following is an overview of the PLM specification. The specification includes classes: Part, PartUsage, LogicalDesign and PhysicalDesign, and Dimension; links between these within the PLM domain and links to classes in other OSLC domains. The specification also defines the PLM effectivity model: Effectivity, EffectivityContext, the oslc_plm:EffectivitySelections specialization of oslc_config:Selections, the oslc_plm:effectivityContext extension property on oslc_config:Configuration, and the algorithm for computing EffectivitySelections.selects (see Effectivity). Variability — the orthogonal configuration-option dimension — is not part of this specification; it is defined by the OSLC Variability specification under specs/core/.

Resource Formats

In addition to the requirements for resource representations in [[!OSLCCore3]], this section outlines further refinements and restrictions.

For HTTP GET/PUT/POST requests on all OSLC PLM and OSLC Core defined resource types,

  • PLM Servers MUST support RDF/XML representations with media-type application/rdf+xml. PLM Clients SHOULD be prepared to deal with any valid RDF/XML document.
  • PLM Servers MUST support XML representations with media-type application/xml. The XML representations MUST follow the guidelines outlined in the OSLC Core Representations Guidance to maintain compatibility with [[!OSLCCore2]].
  • PLM Servers MAY support JSON representations with media-type application/json. The JSON representations MUST follow the guidelines outlined in the OSLC Core Representations Guidance to maintain compatibility with [[!OSLCCore2]].

Resource Operations

For compatibility with OSLC 2.0, OSLC PLM Servers MAY accept the OSLC Core Version header (OSLC-Core-Version: 2.0) in any HTTP request as specified in [[!OSLCCore3]], and return an OSLC PLM 2.0 representation (including the OSLC-Core-Version: 2.0 header). If the OSLC Core Version header is absent on a request, or has some undefined value, the OSLC PLM Server MUST return an PLM 2.0 representation.

OSLC PLM Servers MUST support HTTP GET requests on PLM Resources, with an Accept header of application/rdf+xml, and return the RDF/XML representation of the resource.

OSLC PLM Servers SHOULD support HTTP GET requests on PLM Resources, with an Accept header of an HTML type ( application/html, application/xhtml), and return either an HTML/XHTML representation of the resource or redirect the client to another URL that can (i.e. 302 Redirect).

OSLC PLM Servers SHOULD support HTTP GET requests for user interface (UI) preview of PLM Resources as defined by [[!OSLCPreview]].

OSLC PLM Servers SHOULD support resource modifications on PLM Resources with standard HTTP PUT and DELETE methods. PLM Servers MAY limit modifications in any way they want. For example a service provider may limit updates to resources to simple link properties of link types already defined in the provider. Modification methods MUST use the If-Match header for concurrency management. Providers MAY discard such properties and continue a PUT operation without warning to the client.

OSLC PLM Servers SHOULD support resource modifications on LinkType Resources (LTR) with standard HTTP PUT and DELETE methods. PLM Servers MAY limit modifications in any way they want. For example a service provider may not support additional properties. Modification methods SHOULD use the If-Match header for concurrency management.

Requesting and Updating Properties

Requesting a Subset of Properties

An OSLC PLM server MAY support the oslc.properties URL query parameter on an HTTP GET request on individual resource request or a collection of resources by query. If the oslc.properties query parameter is omitted on the request, then all resource properties MUST be provided in the response.

Updating a Subset of Properties

An OSLC PLM client MAY request that a subset of a resource's properties be updated by identifying those properties to be modified using the oslc.properties URL parameter on a HTTP PUT request.

Updating Multi-Valued Properties

An OSLC PLM Server MAY support updating a subset of a resource's properties by using the [[LDPPatch]] PATCH method.

For compatibility with [[!OSLCCore2]], an PLM Server MAY also support partial update by identifying those properties to be modified using the oslc.properties URL parameter on a HTTP PUT request.

If the parameter oslc.properties contains a valid resource property on the request that is not provided in the content, the server MUST set the resource's property to a null or empty value. If the parameter oslc.properties contains an invalid resource property, then a 409 Conflict MUST be returned.

Vocabulary Terms and Constraints

OSLC PLM Resources 2.1 Defines the vocabulary terms and constraints for OSLC Product Lifecycle Management resources. These terms and constraints are specified according to [[!OSLCCore3]].

Effectivity

A resolved Bill of Materials view is, in OSLC terms, a configuration in which the selected oslc_plm:Part and oslc_plm:PartUsage versions have been further refined against the instance / timing parameters under which the product is being built or analysed. This section defines:

  • Effectivity records carried by Part and PartUsage versions, expressing the date / unit / serial / lot / model parameters under which the version applies;
  • The EffectivityContext resource carrying the parameter values supplied to predicate evaluation;
  • The oslc_plm:effectivityContext property on oslc_config:Configuration that identifies which EffectivityContext applies to the configuration's selections;
  • oslc_plm:EffectivitySelections, a specialization of oslc_config:Selections whose oslc_config:selects triples name the Part and PartUsage version resources that have survived effectivity evaluation;
  • The computation algorithm by which a server populates EffectivitySelections.selects from the candidate pool and the effectivity context.

Variability is not covered here. Variability — the configuration-option dimension (feature choices, option values) — is orthogonal to effectivity and is specified by the OSLC Variability specification in specs/core/. A configuration MAY combine PLM effectivity with OSLC variability; this PLM specification does not depend on variability being present.

No changes to OSLC Configuration Management. oslc_plm:EffectivitySelections is a subclass of oslc_config:Selections, and oslc_plm:effectivityContext is a property carried on oslc_config:Configuration as a PLM-domain extension. Both are outgoing references to OSLC CM, not changes to it.

Because oslc_plm:PartUsage is a first-class, versioned reification of the "uses / used-by" relationship between Parts, BOM-line participation is expressed by including or excluding a PartUsage version through the same effectivity mechanism used for Part versions. No separate "effectivity on relationships" mechanism is required.

Effectivity Records, Predicates, and Context

Effectivity Records on Part and PartUsage Versions

A oslc_plm:Part or oslc_plm:PartUsage version MAY carry one or more oslc_plm:Effectivity records via the oslc_plm:effectivity property. Each Effectivity record carries:

  • oslc_plm:effectiveValueType — the type of value the range is expressed in. One of oslc_plm:Date, oslc_plm:UnitNumber, oslc_plm:SerialNumber, oslc_plm:Lot, or oslc_plm:Model.
  • oslc_plm:effectiveForEndItem — the end-item Part defining the serial/unit namespace. REQUIRED for UnitNumber and SerialNumber records, since unit and serial numbers are otherwise ambiguous across end-items. OPTIONAL for the other value types.
  • oslc_plm:effectiveFrom and oslc_plm:effectiveTo — the inclusive bounds of the range. Datatypes match effectiveValueType. Absence of either bound means open-ended.
  • oslc_plm:effectiveReplacement (optional) — when present, the predicate evaluates as a substitution rather than a Boolean. If the supplied context falls within [effectiveFrom, effectiveTo], the candidate version is replaced in the resolved binding by the version named here. This expresses release effectivity / supersede semantics. The target MUST be a different version of the same versioned resource as the candidate.
  • oslc_plm:authorizedBy (optional) — the change-management resource that authorized this record. Provides audit linkage required by many enterprise PLM workflows.

Multiple Effectivity records on a single version are interpreted as a logical OR: the version is effective under the supplied context if any of its records' predicates evaluates to true. This supports the common pattern of one part revision being effective on several end-items with different unit/serial ranges (for example, Aircraft ABC units 1–5 and Aircraft XYZ units 9–11).

Predicate Evaluation

For each Effectivity record carried by a candidate version, the server evaluates:

  1. Context-type match. If the supplied oslc_plm:EffectivityContext does not carry a value of the record's effectiveValueType (for example, the record is a UnitNumber record but the context supplies only a date), the record evaluates to neither true nor false; it is skipped and contributes nothing to the OR of records on this version.
  2. End-item match. If the record specifies effectiveForEndItem, the context's effectivityEndItem MUST match for the record to be considered. Otherwise the record is skipped.
  3. Range comparison. The context value (date, unit number, etc.) is compared to [effectiveFrom, effectiveTo]. Open-ended bounds are admitted as expected.
  4. Outcome. If effectiveReplacement is absent, the record contributes the Boolean result of the range comparison to the OR. If effectiveReplacement is present and the range comparison is true, the record yields the replacement version URI; this preempts the Boolean OR with a substitution outcome (see Computation of EffectivitySelections.selects, Procedure step 2, the version-URI outcome).

If no records contribute a positive outcome (Boolean true or substitution), the version is removed from the resolved binding.

Effectivity Context

An oslc_plm:EffectivityContext resource carries the parameter values supplied to effectivity predicate evaluation. It is referenced from an oslc_config:Configuration via the oslc_plm:effectivityContext extension property defined by this specification (see EffectivitySelections and the effectivityContext Property). The Configuration's URL is the single entry point that identifies all information needed to evaluate effectivity against the configuration's selections; no request-time header or query parameter mechanism is defined or needed.

The shape :EffectivityContextShape permits any combination of: oslc_plm:effectivityDate, oslc_plm:effectivityEndItem, oslc_plm:effectivityUnitNumber, oslc_plm:effectivitySerialNumber, oslc_plm:effectivityLot, oslc_plm:effectivityModel. A context SHOULD supply only the parameters relevant to the predicates it expects the configuration's selected versions to evaluate against; missing parameters cause records of the corresponding type to be skipped as described above.

When an EffectivityContext resource carries effectivityUnitNumber or effectivitySerialNumber, it MUST also carry an effectivityEndItem; a server MUST reject contexts that lack the required end-item with 400 Bad Request.

EffectivitySelections and the effectivityContext Property

This specification defines oslc_plm:EffectivitySelections as a specialization of oslc_config:Selections, and oslc_plm:effectivityContext as an extension property on oslc_config:Configuration. Together they integrate PLM effectivity into the OSLC Configuration Management resolution model without modifying the OSLC CM specification itself.

oslc_plm:EffectivitySelections

An oslc_plm:EffectivitySelections resource is a specialization of oslc_config:Selections whose oslc_config:selects triples reference Part and PartUsage version resources that have already been determined to be effective under the EffectivityContext identified by the containing configuration's oslc_plm:effectivityContext property. The EffectivityContext is not stored on the EffectivitySelections resource itself — it is a property of the containing configuration, ensuring that a baseline (which freezes both its selects triples and its effectivityContext reference) is self-describing.

The target of oslc_config:selects on an EffectivitySelections resource MUST be a version resource. Each targeted version is one that:

  1. has been selected as a candidate by the Configuration-Context of the containing configuration through the version resolution defined in the OSLC CM base specification, Section 12; and
  2. has been evaluated as effective by its oslc_plm:effectivity records against the EffectivityContext identified by the containing configuration's oslc_plm:effectivityContext property.

Targets MUST NOT be of type oslc_config:Configuration or oslc_config:Component. Typical targets are oslc_plm:Part and oslc_plm:PartUsage versions, but any versioned resource carrying oslc_plm:effectivity records is admissible.

In a baseline, this property is read-only and frozen at baseline-creation time. In a stream, this property is server-maintained: the server re-computes its values when either the stream's candidate pool or the stream's oslc_plm:effectivityContext reference changes.

oslc_plm:effectivityContext Property on Configuration

oslc_plm:effectivityContext is a property on oslc_config:Configuration identifying the EffectivityContext applicable to the configuration's EffectivitySelections resources. A configuration whose selections include one or more oslc_plm:EffectivitySelections MUST carry exactly one oslc_plm:effectivityContext property naming the context against which those selections' post-filter selects values were computed.

The property attaches the EffectivityContext to the configuration so that a baseline is self-describing: reading the baseline yields both its frozen selects triples and the context reference that produced them, with no external lookup. The Configuration's URL is therefore the single entry point that identifies all information needed to evaluate effectivity against its selections; no request-time header or query parameter mechanism is defined or needed.

On a baseline, oslc_plm:effectivityContext is read-only and frozen. On a stream, the property is mutable and modification causes re-computation of the affected EffectivitySelections.selects.

If a configuration is created with EffectivitySelections but no oslc_plm:effectivityContext, the server MUST fail the create operation with a 400 Bad Request indicating that an effectivity context is required.

Computation of EffectivitySelections.selects

This section defines the algorithm by which a server populates the oslc_config:selects triples of an oslc_plm:EffectivitySelections resource. The algorithm is invoked when:

  • An EffectivitySelections resource is first created in a stream;
  • The stream's oslc_plm:effectivityContext reference is changed or the referenced context resource is edited;
  • The stream's candidate pool changes (a contributing oslc_config:Selections or oslc_config:UnboundSelections is created, updated, or removed);
  • A baseline is created from the stream — the algorithm runs once to materialize the frozen selects triples that the baseline will record.

A baseline's selects triples MUST NOT be re-computed after baseline creation.

Inputs

The server MUST take as inputs:

  1. The candidate pool — the version resources in scope for the containing configuration's Configuration-Context, as determined by Section 12 version resolution across the configuration's oslc_config:Selections, oslc_config:UnboundSelections, and recursive contributions. Where the OSLC Variability specification's oslc:VariabilitySelections is also present, the candidate pool is restricted to those candidates that also appear in the surviving VariabilitySelections.selects (variability is applied before effectivity).
  2. The EffectivityContext identified by the containing configuration's oslc_plm:effectivityContext property.

Procedure

For each candidate version resource v in the input pool:

  1. If v carries no oslc_plm:effectivity records, include v in EffectivitySelections.selects unchanged.
  2. Otherwise, evaluate each effectivity record carried by v against the EffectivityContext per Predicate Evaluation. Records combine by logical OR: v is effective if any one record yields a positive outcome.
    • Boolean true — include v in EffectivitySelections.selects.
    • Boolean false — the record does not apply; continue to the next record.
    • Version URI (release-effectivity substitution) — include the named replacement version (not v) in EffectivitySelections.selects. The replacement MUST be a version of the same versioned resource as v; otherwise the record is malformed and the server MUST fail the computation with a 500 Internal Server Error.
  3. If every record carried by v evaluates to Boolean false, v is not included in EffectivitySelections.selects.

Cross-Contribution boundaries

Where a configuration is contributed to a higher-level configuration via oslc_config:contribution, the EffectivitySelections.selects of the contributing configuration appears in the higher-level view as an already-computed survivor set. Effectivity is evaluated within each contributing configuration against that configuration's own oslc_plm:effectivityContext; the context does not propagate across oslc_config:contribution boundaries. A higher-level configuration that wishes to impose a single effectivity context across all contributors MUST arrange for each contributing configuration to declare a compatible effectivityContext; this specification does not provide automatic cross-contribution context propagation. See PartUsage Hierarchy vs Cross-Component Composition via Contribution for design guidance on choosing between in-Component PartUsage composition and cross-Component Contribution composition.

Alignment with Industry Practice

The effectivity vocabulary defined in this section is designed to map cleanly to the effectivity concepts used in widely-deployed PLM systems:

  • PTC Windchill. An oslc_plm:Effectivity record corresponds to a Windchill Effectivity object (WTDatedEffectivity, WTSerialNumberedEffectivity, ProductBlockEffectivity, ProductLotNumberEffectivity, ProductSerialNumberEffectivity) attached to a WTPart revision. effectiveForEndItem corresponds to Windchill's EffContext. effectiveReplacement corresponds to the supersedes/supersededBy replacement that an Effectivity ConfigSpec resolves through.
  • Siemens Teamcenter. An oslc_plm:Effectivity record corresponds to an Effectivity object on a ReleaseStatus of an ItemRevision (revision effectivity) or to an occurrence effectivity on a BOM line — handled here via effectivity on a PartUsage version. effectiveForEndItem corresponds to Teamcenter's End Item / Configuration Item.
  • Aras Innovator. An oslc_plm:Effectivity record corresponds to an Aras Effectivity Expression on a Part BOM relationship row.

Alignment with ISO 10303 (STEP) configuration effectivity concepts is a target for a future revision; the STEP effectivity, dated_effectivity, and serial_numbered_effectivity entities are direct counterparts to the value-type identifiers defined here.

Conformance

A PLM Server that claims conformance to this section MUST:

  • Accept oslc_plm:Effectivity resources as values of oslc_plm:effectivity on oslc_plm:Part and oslc_plm:PartUsage versions per the shapes defined in plm-shapes.ttl.
  • Support the oslc_plm:EffectivitySelections resource class and the oslc_plm:effectivityContext property on oslc_config:Configuration as defined in EffectivitySelections and the effectivityContext Property.
  • Implement effectivity predicate evaluation as defined in Predicate Evaluation, including the substitution outcome when effectiveReplacement is present.
  • Implement the computation of EffectivitySelections.selects defined in Computation of EffectivitySelections.selects, including the triggers for re-computation on streams and the freeze on baselines.
  • Freeze a baseline's EffectivitySelections.selects and effectivityContext at baseline creation, and reject subsequent modifications.
  • Reject EffectivityContext resources that carry effectivityUnitNumber or effectivitySerialNumber without effectivityEndItem with 400 Bad Request.

Worked Example — Eco Line Washing Machine

This non-normative worked example illustrates the principal resource classes and relationships defined by this specification — oslc_plm:Part, oslc_plm:PartUsage, the composition properties between them, oslc_plm:Effectivity records on PartUsage versions, and the post-filter oslc_plm:EffectivitySelections mechanism that consumes them. Variability is the orthogonal configuration-option dimension defined by the OSLC Variability specification under specs/core/; it is not exercised here, and a parallel example may be added in a subsequent revision.

The Eco Line washing machine is composed of a Motor, a Housing, and a Control Unit. The Housing is in turn composed of an Upper Housing and a Lower Housing. Each subsystem is an oslc_plm:Part with one or more revisions; each parent→child usage is reified as an oslc_plm:PartUsage whose versions pin specific child revisions and carry the effectivity records that determine which usage is active for a given Effectivity-Context.

                             │                    └─ uses ──> LowerHousing
                             └─ uses ──> ControlUnit

  • em:Usage_WM_Motor_pre27 (effective until 2027-06-30) — ✓
  • em:Usage_WM_Housing_27 (effective 2027) — ✓
  • em:Usage_WM_ControlUnit_v1 (always effective) — ✓

All other PartUsage versions of the same three usages fail effectivity for this context and are absent from selects. The Housing's own configuration (Housing V4.2, pinned by em:Usage_WM_Housing_27) carries unconditional Housing → UpperHousing and Housing → LowerHousing PartUsage versions in its own selections; recursive walking through the configuration's PartUsage versions yields the resolved BOM.

Equivalent turtle for the configuration and its EffectivitySelections would look like: 

em:Stream_EcoLine a oslc_config:Stream ;
  oslc_config:component         em:WashingMachine ;
  oslc_plm:effectivityContext   em:Ctx_2027_06_01 ;

PartUsage Hierarchy vs Cross-Component Composition via Contribution

This informative section addresses a design question that surfaces whenever an OSLC PLM model is assembled into a configuration: a product's structure can be expressed as an oslc_plm:PartUsage hierarchy within a single oslc_config:Component, or as a composition of multiple PLM Components stitched together by OSLC Configuration Management's oslc_config:contribution mechanism. The two hierarchies look superficially similar — both are compositional assembly — but they answer different questions and operate at different scales.

The PartUsage hierarchy

An oslc_plm:Part with oslc_plm:composedOfPartUsage targets defines the product structure within a single PLM Component. The hierarchy is the BOM: each PartUsage version reifies a parent→child usage relationship and may carry oslc_plm:effectivity records (PLM domain) and, where applicable, oslc:variabilityCondition records (from the OSLC Variability specification in specs/core/) that participate in resolution. All Parts and PartUsage versions in the hierarchy are versioned within the same Component and are co-authored as parts of the same product family. The Effectivity-Context (via oslc_plm:effectivityContext) and Variability-Context (via oslc:variabilityContext) attached to the containing oslc_config:Configuration drive computation within the Component over the Component's own PartUsage versions.

Use the PartUsage hierarchy when:

  • The parent Part and its child Parts are co-authored within one PLM Component;
  • The subsystems share a single versioning lifecycle, governance, and effectivity context;
  • The product structure is internal to the product family (no separately-governed supplier or third-party PLM Component is involved);
  • Effectivity and variability should apply uniformly across the structure as a single filter pass.

The Configuration / Contribution hierarchy

An oslc_config:Configuration with oslc_config:contribution targets composes separately-governed configurations, potentially across multiple Components and across multiple OSLC domains. Each Contribution references a local Configuration of some governed source — a PLM Component, an RM module, a QM test plan, an AM architecture model. The global Configuration does not define its contributed configurations; it references them.

Use Configuration / Contribution composition when:

  • Subsystems are separately governed — different teams, different product lines, different versioning lifecycles, possibly different organisations or tools;
  • Each subsystem is its own PLM Component with its own configurations and baselines;
  • The view crosses domains (PLM + RM + QM + AM + CM) — the Configuration hierarchy is the cross-domain assembly surface;
  • Configurations of resources hosted on different OSLC servers must be composed.

The distinguishing rule

Two complementary tests separate the cases:

  1. Governance / authorship boundary. If the parent and child are co-authored within the same Component, use PartUsage. If the child is separately governed, use Contribution.
  2. Identity scope. A PartUsage targets a child Part within the same Component. A Contribution targets a Configuration of a different Component (potentially on a different server).

The two hierarchies layer; they do not compete

In a realistic system both appear, at different scales. The PartUsage hierarchy is the internal anatomy of one PLM Component. Contributions assemble across Components and domains. 

Global Configuration
├── Contribution → RM local Configuration   (Requirements selections)
├── Contribution → QM local Configuration   (Test case selections)
├── Contribution → AM local Configuration   (Architecture model selections)
└── Contribution → PLM local Configuration   ← single PLM Component
                   └── EffectivitySelections.selects → PartUsage versions
                       └── (the PartUsage hierarchy that defines the
                           product structure lives entirely here)

The PLM Component contributes to the global Configuration as a single unit. The internal PartUsage structure of the Component is opaque to the global level: the global Configuration's Contribution references the Component's local Configuration, whose EffectivitySelections.selects names the surviving PartUsage versions. The global Configuration does not impose effectivity or variability across Contribution boundaries; those contexts apply within each contributing Component.

When the choice is real: subsystems of a complex product

A complex product's subsystems can be modelled either way. The choice is a system-engineering decision, not two ways to express the same thing:

  • Subsystems as PartUsages inside one PLM Component. Pick this when the subsystems are co-owned and versioned together, share an effectivity context, and the whole product family is governed as one. The Eco Line washing-machine worked example uses this pattern: Motor, Housing, Upper/Lower Housing, and Control Unit are all PartUsages of one WashingMachine Part within a single Component.
  • Subsystems as separate Components, assembled via Contributions. Pick this when subsystems have independent product lines, separate teams, supplier-provided assemblies, or each subsystem is independently configured for many products. The "Configuration / Configuration Item / Work Product" pattern from MID's Cross Domain Configuration Management is closer to this shape: each subsystem is a distinct Configuration that contributes to the top-level product configuration.

The first models a single configurable product. The second models an assembly of configurable products. Adapter authors and PLM implementers should make this choice deliberately, recognising that the two patterns also compose: a Component that uses the PartUsage pattern internally can still be one Contribution among several in a higher-level global Configuration.

PLM Server Capabilities

Resource Shapes

OSLC PLM servers SHOULD support Resource Shapes as defined in [[!OSLCShapes]].

Service Provider Resources

OSLC PLM Servers MUST provide a ServiceProvider Resource that can be retrieved at a implementation dependent URI.

OSLC PLM Servers MUST provide a ServiceProviderCatalog Resource that can be retrieved at a implementation dependent URI.

OSLC PLM Servers MUST provide an oslc:serviceProvider property for their defined resources that will be the URI to a ServiceProvider Resource. This does not prevent PLM Servers from providing multiple servie provider properties with different values, if the service provider supports multiple OSLC domain specifications, and the resource is applicable to multiple domains.

OSLC PLM Servers MUST supply a value of http://open-services.net/ns/plm# for the property oslc:domain on either oslc:ServiceProvider or oslc:ServiceProviderCatalog resources.

Creation Factories

OSLC PLM Servers MAY support CreationFactories as defined by [[!OSLCCore3]].

OSLC PLM Servers MAY discard properties it does not recognize and continue the POST operation without warning to the client. The returned resource will contain the accepted properties (and server generated properties like the dcterms:identifer) so clients will be able to confirm if required what was accepted.

If OSLC PLM Servers support the creation of resources from the OSLC defined oslc_plm:Part format, there MUST be at least one Creation Factory entry in the Services definition, and its oslc:usage property MUST be set to http://open-services/ns/core#default. The oslc:resourceType MUST be set to http://open-services.net/ns/plm#Part.

If OSLC PLM Servers support the creation of resources from a resource other than oslc_plm:Part, there MUST be a separate creation services definition whose oslc:usage property MUST NOT be set to http://open-services/ns/core#default.

Query Capabilities

OSLC PLM Servers SHOULD support the Query Capabilities as defined by [[!OSLCCore3]] for all PLM resources.

If the service provider supports query capability for PLM resources, it MUST support the following query parameters:

  • oslc.where
  • oslc.searchTerms

Delegated UIs

OSLC PLM Servers SHOULD support the selection of resources by delegated selection dialogs as defined by [[!OSLCCore3]].

OSLC PLM Servers MAY support the creation of resources by delegated creation dialogs as defined by [[!OSLCCore3]].

In oslc:Dialog elements, the two optional child elements; oslc:hintWidth and oslc:hintHeight specify the suggested size of the dialog or frame to render the HTML content in. Expected size values are defined by CSS length units.
