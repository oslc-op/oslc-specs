This specification defines an OSLC vocabulary and resource model for expressing configuration-option variability over OSLC resources, and a specialization of oslc_config:Selections — oslc:VariabilitySelections — whose oslc_config:selects triples name the version resources that have survived variability evaluation against a Variability-Context attached to the containing configuration. The variability predicate (a boolean expression over option-value equality) is applicable to any versioned OSLC resource, regardless of domain.

Introduction

A configurable product, requirement set, or any other family of resources may admit many variants — each variant produced by a choice of option values (Engine = V6 vs Engine = V8, Market = EU vs Market = NA, Tier = Premium vs Tier = Standard, and so on). The set of all candidate resources spanning all possible variants is sometimes called a 150% structure. The set of resources that actually applies under a specific choice of option values is the corresponding 100% structure.

This specification provides a vocabulary for expressing the 150% structure (Options, OptionValues, OptionSets, and boolean VariabilityConditions carried by candidate versions) plus the mechanism by which the OSLC Configuration Management base specification's oslc_config:Selections is specialised to name the 100% survivors: oslc:VariabilitySelections.

Variability in this specification is orthogonal to effectivity. Variability addresses option choice ("Engine = V6"); effectivity addresses instance/timing ("unit 100 onwards"). The OSLC PLM specification's oslc_plm:Effectivity is the canonical effectivity mechanism in OSLC and is a peer to the variability mechanism defined here. A single configuration MAY combine both.

No changes to OSLC Configuration Management. This specification does not modify the OSLC Configuration Management base specification. oslc:VariabilitySelections is a subclass of oslc_config:Selections, and oslc:variabilityContext is an extension property carried on oslc_config:Configuration — both are outgoing references to OSLC CM, not changes to it.

References

VariabilitySelections

oslc:VariabilitySelections is a specialization of oslc_config:Selections whose oslc_config:selects triples name the version resources that have already been determined to apply under the Variability-Context identified by the containing configuration's oslc:variabilityContext property. The Variability-Context is not stored on the VariabilitySelections resource itself; it is a property of the containing configuration, ensuring that a baseline (which freezes both its selects triples and its variabilityContext reference) is self-describing.

VariabilitySelections Resource Constraints

selects semantics

The target of oslc_config:selects on a VariabilitySelections resource MUST be a version resource. Each targeted version is one that:

  1. has been selected as a candidate by the Configuration-Context of the containing configuration through the version resolution defined in the OSLC CM base specification, Section 12; and
  2. has been evaluated as variant-applicable by its oslc:variabilityCondition against the Variability-Context identified by the containing configuration's oslc:variabilityContext property.

Targets MUST NOT be of type oslc_config:Configuration or oslc_config:Component. Targets MAY be of any versioned resource type that carries variability conditions; this specification places no domain restriction on the targets.

In a baseline, this property is read-only and frozen at baseline-creation time. In a stream, this property is server-maintained: the server re-computes its values when either the stream's candidate pool or the stream's oslc:variabilityContext reference changes.

Variability Conditions on Candidate Version Resources

Candidate version resources that participate in variability filtering carry a oslc:variabilityCondition — a boolean expression over the equality of named Options to OptionValues — which the server evaluates against the Variability-Context attached to the containing configuration.

A oslc:VariabilityCondition is either a terminal (oslc:optionEquals) or an internal node combining sub-conditions (oslc:and, oslc:or, oslc:not). A version with no variabilityCondition is treated as unconditionally present (the condition evaluates to true for every VariabilityContext).

A version resource carries at most one oslc:variabilityCondition per version; use AND/OR composition to express compound conditions.

Predicate Evaluation

For each candidate version v in the input pool:

  1. If v carries no oslc:variabilityCondition, include v in VariabilitySelections.selects.
  2. Otherwise, evaluate the condition against the Variability-Context:
    • Boolean true — include v in VariabilitySelections.selects.
    • Boolean false — exclude v.
    • Three-valued (unassigned terminal) — terminals that reference Options not assigned in the VariabilityContext evaluate to neither true nor false; the surrounding boolean handles this per three-valued logic (true OR ? = true, false AND ? = false, otherwise the candidate is retained pending fuller resolution).
    • Any other value — the predicate is malformed; the server MUST fail the computation with a 500 Internal Server Error referencing an oslc:Error describing the failure. [VAR-VC-EVAL]

Variability predicates are not expected to yield substitute version URIs; the question variability answers is "is this version included?" and not "which sibling version applies?". Substitution between revisions based on lifecycle progression is the domain of effectivity (see the OSLC PLM specification's oslc_plm:Effectivity), not variability.

Variability Context

A Variability-Context is a resource that carries the OptionValue choices supplied to variability predicate evaluation. It is referenced from an oslc_config:Configuration via the oslc:variabilityContext extension property defined by this specification.

A configuration whose selections include one or more oslc:VariabilitySelections MUST carry exactly one oslc:variabilityContext property naming the context against which those selections' post-filter selects values were computed. [VAR-CTX-REQ]

The property attaches the variability context to the configuration rather than to individual VariabilitySelections resources so that a baseline is self-describing: reading the baseline yields both its frozen selects triples and the context reference that produced them, with no external lookup. The Configuration's URL is therefore the single entry point that identifies all information needed to evaluate variability against its selections; no request-time header or query parameter mechanism is defined or needed.

Computation of VariabilitySelections.selects

This section defines the algorithm by which a server populates the oslc_config:selects triples of an oslc:VariabilitySelections resource. The algorithm is invoked when:

  • An oslc:VariabilitySelections resource is first created in a stream;
  • The stream's oslc:variabilityContext reference is changed or the referenced context resource is edited;
  • The stream's candidate pool changes (a contributing oslc_config:Selections or oslc_config:UnboundSelections is created, updated, or removed);
  • A baseline is created from the stream — the algorithm runs once to materialize the frozen selects triples that the baseline records.

A baseline's selects triples MUST NOT be re-computed after baseline creation. [VAR-BL-FROZEN]

Inputs

The server MUST take as inputs:

  1. The candidate pool — the version resources in scope for the containing configuration's Configuration-Context, as determined by Section 12 version resolution across the configuration's oslc_config:Selections, oslc_config:UnboundSelections, and recursive contributions.
  2. The Variability-Context identified by the containing configuration's oslc:variabilityContext property.

Procedure

For each candidate version v in the input pool, evaluate the oslc:variabilityCondition carried by v as defined in §Predicate Evaluation, and include v in VariabilitySelections.selects if and only if the result is Boolean true (or v carries no condition). The resulting set of version URIs is the oslc_config:selects value of the VariabilitySelections.

Interaction with effectivity

When the containing configuration also carries oslc_plm:effectivityContext (per the OSLC PLM specification) and contains an oslc_plm:EffectivitySelections, variability is computed first; the surviving VariabilitySelections.selects set defines the candidate pool that effectivity computation then refines. This composition order is normative: effectivity records on a version MAY presuppose option choices that variability has already resolved, but the reverse is not expected. [VAR-VS-EFF-ORDER]

Cross-Contribution boundaries

Where a configuration is contributed to a higher-level configuration via oslc_config:contribution, the VariabilitySelections.selects of the contributing configuration appears in the higher-level view as an already-computed survivor set. Variability is evaluated within each contributing configuration against that configuration's own oslc:variabilityContext; the context does not propagate across oslc_config:contribution boundaries. A higher-level configuration that wishes to impose a single variability context across all contributors MUST arrange for each contributing configuration to declare a compatible variabilityContext; this specification does not provide automatic cross-contribution context propagation.

Caching, Tracked Resource Sets, and Compact Rendering

  • A baseline's stored VariabilitySelections.selects triples are effectively a server-side cache of the resolution; no additional response-cache keying is needed beyond the configuration URI itself.
  • A stream's stored VariabilitySelections.selects is refreshed on context or candidate-pool change.
  • Tracked Resource Sets (per the OSLC Core TRS specification) only track version resources. It is the responsibility of the TRS consumer to provide an appropriate Configuration-Context — which carries with it the configuration's attached variabilityContext — to resolve tracked version resources to a selected version.
  • Compact rendering of resources resolved through VariabilitySelections MUST reflect the variability context attached to the containing configuration at the time of the initial request.

Conformance

A configuration server claiming conformance to this specification MUST:

  1. Support oslc:VariabilitySelections resources in configurations as defined in §VariabilitySelections.
  2. Support the oslc:variabilityContext property on oslc_config:Configuration as defined in §Variability Context, enforcing the read-only/frozen behaviour for baselines and the mutable behaviour for streams.
  3. Implement the computation of VariabilitySelections.selects defined in §Computation of VariabilitySelections.selects.
  4. Freeze a baseline's VariabilitySelections.selects and variabilityContext at baseline creation, and reject subsequent modifications.

A configuration server MAY claim conformance to the OSLC Configuration Management base specification without conforming to this specification. Servers that do not conform MUST reject any configuration POSTed to them that contains oslc:VariabilitySelections resources or an oslc:variabilityContext property with a 400 Bad Request response.

Open Questions for the OASIS OSLC-OP TC

  1. Composition order with effectivity. §Computation specifies variability-before-effectivity binding when an oslc_plm:EffectivitySelections is also present in the configuration. Existing PLM systems differ in framing (Windchill applies the option filter first, then ConfigSpec for revisions; Teamcenter treats Revision Rule and Variant Rule as independent; Aras runs Configurator Services and Effectivity Services separately). Is the variability-first order the right normative choice, or should the spec leave it to domain specs?
  2. Predicate validation. The spec says malformed predicate evaluations result in 500 errors. Should there be a lightweight authoring-time conformance check that domain specifications can require, so malformed predicates are caught at creation rather than at computation?
  3. VariationPoint / VariantMembership layer. SysML v2 supports an explicit variation-point model (isVariation + VariantMembership). Should this specification optionally provide a thin variation-point layer over the existing variabilityCondition so SysML v2 VariantMembership structures can round-trip? (Reference and motivation: see the PLM mapping doc's SysML v2 prior-art section in the PLM specification.)
